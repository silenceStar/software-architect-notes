# 19.4 Kappa 架构

## 1. 本节核心思想（100字以内）

Kappa 架构删除 Lambda 的批处理层，统一使用流处理引擎处理实时与历史数据；数据以不可变、追加日志形式长期保存，业务逻辑变化时通过重放历史数据重新计算结果，从而统一代码与数据口径。

```text
实时数据 ──→ 消息队列 ──→ 流计算引擎 ──→ 结果存储 ──→ 业务查询
                 ↑
          历史数据重放
```

## 2. 考试高频考点

* **考点1：数据具有 When 和 What 两个特性**
  When 表示数据具有发生时间和先后顺序，What 表示已经发生的数据是不可变的事实，修改和删除也应通过追加新事件表达。

* **考点2：Kappa 采用单一流处理链路**
  Kappa 删除 Lambda 的 Batch Layer，通过改进 Speed Layer，使流处理引擎既处理实时数据，也能通过历史重放完成重新计算。

* **考点3：历史数据重算流程**
  修改业务逻辑后启动新作业，从最早可用 Offset 重新消费历史数据，写入新视图；新作业追上实时进度后切换查询，再停止旧作业。

* **考点4：Kappa 的优缺点**
  优点是实时与离线代码统一、数据口径一致、链路简单；缺点是历史日志存储成本高、回放资源消耗大、多流关联和乱序处理困难。

* **考点5：Kappa+ 架构**
  Kappa+ 使用 HDFS、Hudi 等数据湖保存历史数据，历史回算时由流计算框架直接读取数据湖，避免在消息队列中长期保存全部日志。

* **易混点：Kappa 并不是不处理历史数据**
  Kappa 放弃的是独立的批处理层，而不是历史计算；它通过消息日志或数据湖中的历史数据重放完成 Backfill。

* **易混点：Kappa 不是 Lambda 的完全替代方案**
  Kappa 是 Lambda 的简化架构，适合增量写入、时序数据和固定逻辑的实时计算；Lambda 更适合复杂、灵活的历史数据探索分析。

* **易考选择题：关于 Kappa 架构的正确描述**
  正确选项通常是“删除批处理层”“使用一套流处理代码”“通过历史数据重放重新计算”；“完全不保存历史数据”或“历史数据仍由独立批处理层计算”均错误。

## 3. 记忆口诀

### 口诀

> **一条流、全追加、能重放、换视图；存储贵、关联难，Kappa+用数据湖补。**

### 解释

* **一条流**：只保留一套流处理链路；
* **全追加**：历史事实不可变，通过追加事件记录变化；
* **能重放**：业务逻辑变化后重新消费历史数据；
* **换视图**：新结果追上实时进度后切换旧视图；
* **存储贵**：消息队列长期保存历史日志成本较高；
* **关联难**：多实时流关联容易受延迟和乱序影响；
* **数据湖补**：Kappa+使用 HDFS、Hudi 等保存历史数据。

## 4. 考试一句话总结（精简）

> **最可能考 Kappa 删除批处理层、统一流处理代码，并通过历史数据重放完成重新计算，以及它与 Lambda、Kappa+ 的区别。**
