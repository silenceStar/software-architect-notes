# 19.6 大数据架构设计案例分析

## 1. 本节核心思想（100字以内）

通过奥运数据、广告分析、证券日志和电商决策案例，说明 Lambda 架构适合兼顾历史全量计算与实时增量处理，Kappa 架构适合以统一流处理引擎完成实时计算；实际架构还应通过计算前移、结果预聚合和合理选型降低服务层复杂度。

## 2. 考试高频考点

* **考点1：Lambda 架构的案例应用**
  奥运平台通过批处理层计算完整历史数据、速度层计算实时增量数据，再由服务层合并 Batch View 和 Real-time View，以兼顾准确性和低延迟。

* **考点2：Kappa 架构的案例应用**
  证券日志和电商决策平台以 Kafka、Flink 为核心，使用统一流处理链路完成日志解析、指标计算、实时监控和模型参数更新。

* **易混点：Kappa 架构不是不处理历史数据**
  Kappa 取消的是独立的批处理代码链路，历史数据仍可通过消息重放进入同一流处理系统重新计算。

* **易考选择题：广告平台三版架构演进的核心是什么**
  核心是将关联、聚合和离线实时合并等复杂计算，从查询服务层逐步前移到数据采集与计算阶段，最终使服务层只需读取统一结果表。

### 架构案例对比图

```text
Lambda 架构
历史数据 ──→ 批处理层 ──→ Batch View ──┐
                                         ├─→ 合并查询
实时数据 ──→ 速度层 ────→ Real-time View ┘

Kappa 架构
实时数据/历史重放 ──→ Kafka ──→ Flink ──→ 存储与应用
```

## 3. 记忆口诀

**口诀：奥广双层合，证商一流过；计算往前移，查询一表得。**

**解释：**

* **奥广双层合**：奥运平台和广告平台采用 Lambda 思想，批处理结果与实时结果合并。
* **证商一流过**：证券日志和电商决策采用 Kappa 思想，由统一流处理引擎处理数据。
* **计算往前移**：将关联、聚合等复杂操作提前到采集层或计算层完成。
* **查询一表得**：提前形成统一结果表，减少服务层临时关联和计算压力。

## 4. 考试一句话总结（精简）

本节最可能考 Lambda 与 Kappa 的案例选型、处理流程，以及广告平台通过计算前移降低查询服务复杂度的演进思想。
