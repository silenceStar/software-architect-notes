# 13.3 中间层架构设计

## 1. 本节核心思想（100字以内）

中间层（业务逻辑层）是系统的核心，负责实现业务规则、事务管理、工作流控制和业务实体管理，通过 Service、DAO、Facade、工作流及业务框架（Domain Model–Service–Control）实现业务与表现层、数据层解耦，提高系统的复用性、扩展性和可维护性。

---

## 2. 考试高频考点（精简）

- **考点1：业务逻辑组件设计**（Service=接口+实现类，调用多个DAO，实现业务逻辑与事务管理。）
- **考点2：Spring IoC/DI**（DAO由Spring容器注入，实现依赖注入，降低组件耦合。）
- **考点3：Facade模式**（向Web层提供统一业务接口，隐藏内部多个Service，实现松耦合。）
- **考点4：工作流五大接口**（①流程定义②客户端③应用调用④工作流协作⑤管理监控，是高频选择题。）
- **考点5：业务实体表示方式**（XML、通用DataSet、有类型DataSet，比较各自优缺点。）
- **考点6：DSC业务框架**（Domain Model负责数据，Service负责业务，Control负责流程控制。）

- **易混点**
  - MVC（页面组织）≠ DSC（业务组织）
  - DAO负责数据访问，Service负责业务逻辑
  - Facade是统一业务入口，不是DAO

- **易考选择题**
  - 工作流五个接口分别负责什么功能
  - Service、DAO、Facade各自职责
  - XML、DataSet、有类型DataSet优缺点比较
  - Domain、Service、Control三者职责

---

## 3. 记忆口诀（精简）

**口诀：**

> **接口实现写业务，Spring注入降耦合；Facade统一入口，工作流五接口；DSC管业务，DAO只查库。**

**解释：**

- 接口+实现：业务组件设计模式
- Spring注入：IoC/DI降低耦合
- Facade：统一业务入口
- 工作流：牢记五大接口
- DSC：业务框架核心思想
- DAO：仅负责数据库访问

---

## 4. 考试一句话总结（精简）

**中间层重点考 Service职责、Spring IoC、Facade模式、工作流五接口以及 Domain–Service–Control（DSC）业务框架。**
