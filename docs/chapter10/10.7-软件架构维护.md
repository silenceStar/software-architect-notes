# 10.7 软件架构维护

## 1. 本节核心思想（100字以内）

软件架构维护是保障架构持续演化的重要活动，主要包括架构知识管理、架构修改管理和架构版本管理。通过记录架构设计决策、控制修改影响范围、管理架构演化版本，保证系统在需求变化过程中保持可维护性、可扩展性和可追溯性。

---

## 2. 考试高频考点（尽量精简）

### （1）架构知识管理
- 架构知识 = 架构设计 + 架构设计决策
- 记录“是什么”及“为什么这样设计”
- 防止架构知识沉没和腐蚀

### （2）架构修改管理
- 核心思想：控制修改影响范围
- 建立隔离区域（RoQ，Region of Quiescence）
- 减少波及效应和副作用

### （3）架构版本管理
- 管理架构演化过程
- 支持版本控制、评价和追踪
- 为架构度量提供依据

### （4）架构可维护性度量指标（了解）
- CCN：圈复杂度（复杂度）
- FFC：扇入扇出度（关联度）
- CBO：模块间耦合度（耦合度）
- RFC：模块响应（功能调用复杂度）
- TCC：紧内聚度（内聚性）
- LCC：松内聚度（内聚性）

### （5）可维护性原则
- 高内聚、低耦合
- CBO低、TCC/LCC高

## 3. 记忆口诀（尽量精简）

### 架构维护三大内容
> 知识、修改、版本

### 架构知识定义
> 架构知识 = 设计 + 决策

### 修改管理
> 建隔离区，控波及效应

### 六大度量指标
> 复杂关联耦合响，紧松内聚要记牢

对应：
- CCN → 复杂度
- FFC → 关联度
- CBO → 耦合度
- RFC → 响应度
- TCC → 紧内聚
- LCC → 松内聚

### 可维护性原则
> 高内聚，低耦合
