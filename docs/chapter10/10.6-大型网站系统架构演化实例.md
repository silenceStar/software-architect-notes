# 10.6 大型网站系统架构演化实例

## 1. 本节核心思想（100字以内）

大型网站架构并非一次设计完成，而是随着用户规模、访问量和数据量增长不断演化。其核心思想是：发现系统瓶颈→拆分瓶颈→引入新技术解决问题，逐步实现高并发、高可用、高性能、高扩展和易维护的系统架构。

---

## 2. 考试高频考点（尽量精简）

### 架构演化顺序（必考）

1. 单体架构
2. 应用与数据分离（垂直架构）
3. 缓存
4. 应用服务器集群
5. 数据库读写分离
6. CDN + 反向代理
7. 分布式文件系统 + 分布式数据库
8. NoSQL + 搜索引擎
9. 业务拆分
10. 分布式服务（SOA/微服务）

---

### 各阶段解决的问题（常考）

| 阶段 | 解决问题 |
|--------|--------|
| 缓存 | 数据库读压力 |
| 集群 | 应用服务器并发压力 |
| 读写分离 | 数据库整体压力 |
| CDN | 跨地域访问慢 |
| 反向代理 | 减轻后端压力 |
| 分布式存储 | 单机存储瓶颈 |
| NoSQL | 海量非结构化数据 |
| 搜索引擎 | 全文检索需求 |
| 业务拆分 | 系统复杂度过高 |
| 分布式服务 | 服务复用与治理 |

---

### 常见对应技术（选择题）

| 需求 | 技术 |
|--------|--------|
| 缓存 | Redis |
| 负载均衡 | Nginx |
| 读写分离 | MySQL主从 |
| CDN加速 | CDN |
| 分布式文件 | HDFS、FastDFS |
| NoSQL | Redis、HBase、MongoDB |
| 搜索引擎 | Elasticsearch |
| 分布式服务 | SOA、微服务、Dubbo、Spring Cloud |

### 瓶颈记忆法（案例分析题）

```text
数据库慢 → 缓存
应用扛不住 → 集群
数据库太忙 → 读写分离
访问太慢 → CDN
存储不够 → 分布式存储
检索困难 → 搜索引擎
业务太复杂 → 拆分
系统太多 → 服务化
```
