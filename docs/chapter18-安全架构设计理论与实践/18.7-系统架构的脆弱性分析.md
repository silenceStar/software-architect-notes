# 18.7 系统架构的脆弱性分析

## 1. 本节核心思想（100字以内）

系统安全遵循“木桶理论”，整体安全性取决于最薄弱环节。脆弱性可能来源于设计、实现、软件缺陷及错误配置，并贯穿物理设备、软件、网络、人员和管理制度。不同软件架构因结构特点不同，具有不同的脆弱性，应持续进行发现、分析、修补和防护。

## 2. 考试高频考点

- **考点1：脆弱性的主要来源**  
  主要包括软件设计瑕疵、软件实现弱点、软件自身缺陷以及系统和网络错误配置。

- **考点2：软件脆弱性的生命周期**  
  脆弱性经历“引入—产生破坏效果—修补—消失或风险降低”的过程，其管理是一项持续性工作。

```text
脆弱性引入
    ↓
被攻击者发现或利用
    ↓
非法执行代码、篡改数据、非法访问、拒绝服务
    ↓
检测与修补
    ↓
漏洞消失或风险降低
````

* **考点3：软件脆弱性的特点**
  脆弱性具有隐蔽性、逻辑错误根源性、环境相关性和长期存在性，本身不一定造成危害，被利用后才会形成安全后果。

* **考点4：脆弱性分类方法**
  Landwehr分类法从缺陷的起因、引入时间和分布位置三个维度分类；IBM分类法在其基础上扩展，面向脆弱性检测工具开发。

* **考点5：脆弱性扫描方法**
  基于特征的扫描又称基于知识的扫描或违规扫描，通过与漏洞特征库比对发现已知漏洞，其效果取决于特征库的完整性和更新程度。

* **考点6：典型架构的脆弱性**

| 架构类型   | 主要脆弱性                       |
| ------ | --------------------------- |
| 分层架构   | 底层故障向上传播，层间通信和调用开销较大        |
| C/S架构  | 客户端容易被分析，账号密码和通信数据可能被截取     |
| B/S架构  | 系统暴露在开放网络中，容易受到Web和HTTP相关攻击 |
| 事件驱动架构 | 事件顺序难控制，可能出现死循环、高并发和数据不一致   |
| MVC架构  | 结构复杂，视图与控制器耦合，模型数据访问效率较低    |
| 微内核架构  | 难以整体优化，插件或进程间通信开销较大         |
| 微服务架构  | 分布式通信、局部故障、服务管理和数据一致性复杂     |

* **易混点：软件缺陷与软件脆弱性**
  软件缺陷不一定形成脆弱性；只有缺陷能够被攻击者利用并导致安全策略被违反时，才构成安全脆弱性。一个脆弱性可能由多个缺陷共同导致。

* **易混点：设计缺陷与实现缺陷**
  设计缺陷是系统方案或安全规则本身存在问题；实现缺陷是设计方案正确，但程序编码或具体实现出现错误。

* **易考选择题：Landwehr分类法的三个维度**
  分别是缺陷的起因、缺陷引入的时间和缺陷分布的位置，可记为“因、时、位”。

* **易考选择题：事件驱动架构的典型风险**
  包括组件执行顺序不确定、组件间数据交换困难、逻辑关系复杂、事件死循环以及高并发下的数据错误或丢失。

## 3. 记忆口诀

### 口诀一：漏洞来源

**“设计有瑕疵，实现有弱点，软件有缺陷，配置出风险。”**

解释：

* 设计瑕疵：协议、安全模型或系统方案本身不合理；
* 实现弱点：编码过程出现权限、边界或逻辑错误；
* 软件缺陷：输入不校验、异常不处理、资源不控制；
* 配置风险：默认密码、错误开放端口、权限设置过大。

### 口诀二：架构脆弱性

**“层怕传，C怕端，B怕网，事件怕乱，MVC怕繁，内核怕通信，微服务怕分布。”**

解释：

* 分层架构怕故障层层传播；
* C/S架构怕客户端被逆向和数据被截取；
* B/S架构怕开放网络攻击；
* 事件驱动架构怕乱序、循环和并发；
* MVC架构怕结构复杂和调用低效；
* 微内核架构怕插件与进程通信开销；
* 微服务架构怕分布式通信和服务治理复杂。

### 口诀三：脆弱性分析

**“查根源、看影响、懂利用、能修补、防攻击、会探测、可预测。”**

解释：脆弱性分析不仅要发现漏洞，还要分析产生原因、危害、攻击方式、修补方法、防护措施以及未来可能出现的新漏洞。
