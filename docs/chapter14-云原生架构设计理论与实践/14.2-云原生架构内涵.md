# 14.2 云原生架构内涵（加强版）

## 1. 本节核心思想（100字以内）
云原生架构通过云平台能力（IaaS/PaaS）承接高可用、弹性、容灾、可观测、安全等非功能性能力，将应用从“全功能自实现”转变为“只关注业务逻辑”，并通过服务化、自动化与云服务解耦，实现轻量化、弹性化和持续交付的软件架构体系。

---

## 2. 考试高频考点

- **考点1（云原生本质）**  
  云原生的核心是“非功能性能力云化”，即高可用、扩缩容、安全、监控等由云平台承担，应用只保留业务逻辑。

- **考点2（代码结构变化）**  
  云原生应用由三部分构成：业务代码 + 三方库 + 非功能性代码，其中非功能性能力逐步被云平台或中间件下沉。

- **考点3（Mesh架构）**  
  Mesh（Service Mesh）将RPC、缓存、MQ等SDK从业务进程剥离，由Sidecar统一处理流量控制、熔断、限流、安全等能力，实现“业务与基础设施解耦”。

- **考点4（Serverless特点）**  
  Serverless由云负责运行与调度，按事件触发执行，适用于短时、无状态、事件驱动任务；不适合长时间运行或有状态应用。

- **考点5（可观测三要素）**  
  Logging（日志）+ Tracing（链路追踪）+ Metrics（指标），用于定位分布式系统问题与SLO分析。

- **易混点**  
  - 云原生 ≠ 容器/K8S（只是实现手段）  
  - Mesh ≠ 微服务（Mesh是基础能力下沉）  
  - Serverless ≠ 所有应用（有状态/长任务不适合）  

- **易考选择题**
  - 哪些属于非功能性能力（弹性、容灾、监控等）  
  - Mesh架构特点（Sidecar、SDK下沉）  
  - Serverless适用/不适用场景  
  - 可观测三件套  
  - 云原生目标（业务与非功能解耦）

## 3. 记忆口诀

**口诀：**  
云剥非功三层化，服务解耦靠Mesh，  
事件驱动Serverless，可观测性三件套

**解释：**  
云原生的核心是“非功能下沉 + 服务化 + Mesh解耦 + 事件驱动 + 可观测体系”。

---

## 4. 考试一句话总结（精简）
云原生的本质是通过服务化与Mesh等架构手段，将非功能性能力下沉到云平台，实现业务代码轻量化与系统自动化运行。
