# 20.4 常见问题及解决办法

## 1. 本节核心思想

论文失分通常不是因为完全不懂技术，而是由于审题不准、项目不具体、内容空泛、结构混乱或篇幅失控。写作时应紧扣试题设问，以近年真实项目为主线，选择2～3项关键措施深入展开，并通过具体场景、实施过程和实际效果体现实践经验。

---

## 2. 考试高频考点

* **考点1：必须逐一回应试题设问**
  不能只看论文标题套用固定模板，应认真分析题目中的多个问题，并按照设问顺序组织正文，否则即使内容相关，也可能因答非所问而被判为走题。

* **考点2：篇幅要合理分配**
  摘要通常控制在300～400字，正文控制在2000～3000字；机考虽然不再按答题格计数，但仍应控制篇幅，避免摘要过短、正文不足或内容严重超量。

* **考点3：摘要必须覆盖正文核心内容**
  摘要应包含项目背景、本人职责、采用的主要方法、实施效果和总结，做到阅卷人员即使不看正文，也能了解论文的主要内容。

* **考点4：技术措施宜少而精**
  不要罗列大量架构模式、技术和方法，而应挑选最有代表性的2～3项措施，说明为什么采用、如何实施、遇到什么问题以及取得了什么效果。

* **考点5：所有理论必须落到具体项目**
  系统架构设计师论文不是单纯的知识点默写，而是项目经验总结。每项技术措施都应结合实际业务场景、系统模块、数据规模、性能指标或实施过程进行说明。

* **考点6：必须明确主题项目和个人角色**
  论文应说明项目时间、项目名称、建设背景、主要功能、系统规模，以及本人担任的角色和承担的具体工作，不能只写“参与过某类系统建设”。

* **考点7：主题项目应尽量选择近3年的项目**
  年代过于久远的项目容易给人经验陈旧或套用旧论文的印象，应优先选择近期、真实且自己熟悉的项目。

* **考点8：表达应正式但不能过于僵硬**
  应使用书面语言，可适当使用“我们”“项目组”“本人负责”等表达；避免口语化，也不要全文采用“大一二三、小123”的机械条目结构。

* **易混点：以自己为中心不等于全文只写“我”**
  论文需要突出本人承担的职责和贡献，但项目通常由团队完成。可以使用“本人负责”“我组织完成”，也可以使用“我们采用”“项目组实施”，二者并不矛盾。

* **易混点：写得多不等于写得深**
  罗列十种技术属于“广而浅”，选择三种关键措施并写清背景、决策、实施和效果才属于“有深度”。

* **易考选择题：系统架构设计师论文最致命的问题是什么**
  缺少具体主题项目、没有说明本人角色，或者没有回答题目中的具体设问，通常都属于严重失分问题。

## 3. 常见问题与解决办法

| 常见问题   | 主要表现                 | 解决办法                          |
| ------ | -------------------- | ----------------------------- |
| 走题     | 只看标题，不回答题目中的具体问题     | 将每个设问转换成正文一级或二级结构，写完后逐项核对     |
| 字数不够   | 摘要过短、正文内容单薄          | 提前规划各部分篇幅，补充项目背景、实施过程、问题处理和效果 |
| 字数偏多   | 背景过长、知识点堆砌、措施数量过多    | 压缩理论介绍，保留2～3项核心措施，删除与题目无关内容   |
| 摘要归纳不当 | 只有背景和意义，没有具体工作       | 摘要写清项目、职责、方法、效果和总结            |
| 内容深度不足 | 每项技术只解释概念，没有实施细节     | 采用“问题—分析—方案—实施—效果”展开          |
| 缺少特色   | 内容像教材，任何项目都能套用       | 加入业务场景、系统规模、关键指标、技术选型和实际问题    |
| 口语化严重  | 大量使用“我觉得”“然后”“搞了”等表达 | 改用“经分析”“项目组采用”“实施结果表明”等书面语言   |
| 表达能力较弱 | 语句不通顺、逻辑跳跃、重复较多      | 平时按照模板进行限时写作，并积累常用项目表达        |
| 缺少主题项目 | 只讲行业或技术，不介绍具体项目      | 明确项目时间、背景、功能、规模、角色和职责         |
| 项目年代久远 | 使用多年以前的项目            | 优先选择近3年内参与且熟悉的项目              |
| 结构过于死板 | 全文充满编号和条目            | 使用小标题与自然段结合，避免层级过深            |
| 段落过长   | 一个自然段包含多个观点          | 一段集中表达一个观点，机考时也要主动分段，提高可读性    |

## 4. 推荐的措施展开结构

正文中的每项核心措施，可以按照以下逻辑展开：

```text
项目面临什么问题
→ 为什么选择该方法
→ 具体如何设计和实施
→ 实施过程中遇到了什么困难
→ 如何解决这些困难
→ 最终取得了什么效果
```

例如：

```text
由于原有系统在业务高峰期响应时间过长，我们首先对系统调用链和数据库访问情况进行了分析。
分析发现，热点数据被频繁重复查询，是造成数据库压力过大的主要原因。

针对该问题，我们在应用层引入分布式缓存，并采用缓存预热、合理设置过期时间和缓存降级等措施。
同时，为避免缓存击穿，我们对热点数据采用互斥更新机制。

改造后，核心接口平均响应时间由800ms降低至180ms，数据库查询压力下降约60%，系统能够稳定支撑业务高峰访问。
```

## 5. 记忆口诀

### 口诀

**审题要准，项目要真；措施要少，分析要深；结构要清，数据要硬。**

### 解释

* **审题要准**：逐项回答题目中的所有设问，防止走题。
* **项目要真**：写清具体项目、时间、背景、角色和职责。
* **措施要少**：选择2～3项核心措施，不要面面俱到。
* **分析要深**：说明问题、原因、选择、实施、困难和效果。
* **结构要清**：合理使用标题和自然段，避免段落过长或编号过多。
* **数据要硬**：尽量使用响应时间、并发量、可用性、准确率等量化指标证明效果。

---

## 6. 考试一句话总结

本节最可能考查：**识别论文中的常见失分问题，并掌握“紧扣设问、项目真实、措施少而深、结构清晰、效果量化”的改进原则。**
